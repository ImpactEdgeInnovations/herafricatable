Process Map & Developer Specification

Version 2 — production scope · 30-day delivery · 21 July 2026

01Purpose & how to use this document

The Product Vision Document describes what Her Africa Table is and why it matters. This document translates that vision into how it gets built — the modules, the user flows, the data entities, and the sequencing a developer needs to actually ship it.

Each core journey is expressed two ways: a flowchart showing the process end to end, and a step-by-step technical breakdown — screen, user action, system process, data written, and any downstream triggers.

Read this alongside the Product Vision Document, not instead of it — this document assumes that context and focuses purely on build logic.

Production source of truth

The 30-day release is a controlled, event-led launch—not the delivery date for every long-term module. A capability is considered complete only when authorization is enforced in Postgres, failure and empty states exist, mobile and accessibility checks pass, critical paths are tested, operational actions are auditable, and the feature works on localhost, Vercel preview, and production.

Launch loop: registration → identity verification → payment or manual approval → onboarding → event access → member discovery → connection → messaging → ongoing value.

Confirmed platform decisions: responsive web/PWA first; Supabase email OTP for production identity; temporary password access only during pre-SMTP testing; Vercel hosting; Supabase database/auth/storage; provider-neutral payments with Paystack first; audited manual registration as a first-class fallback; rule-based recommendations before AI.

Pre-event networking ("12 women in Kigali are in your industry") is a query against the guest list, filtered by location/industry/interests, run once the guest list reaches a minimum threshold — this can reuse the matching logic from Section 3.5.

3.7 · Membership, Entitlements & BillingMODULE 13
flowchart TD A["Registration request"] --> B{Admin payment mode} B -->|Automatic| C["Paystack transaction initialized"] B -->|Manual review| D["Reference and evidence submitted"] B -->|Closed| E["Waitlist captured"] C --> F["Signed webhook verified and reconciled"] D --> G["Authorized admin approves or rejects"] F --> H["Shared entitlement issuer"] G --> H H --> I["Event and/or membership entitlement created exactly once"] I --> J["Member onboarding enabled"] J --> K{Later renewal} K -->|Success| I K -->|Failure| L["Grace period → dormant, never deleted"]

Launch supports event registration and approved membership through one entitlement model. Automatic Paystack processing and audited manual approval must grant the same entitlement through the same idempotent operation. A browser redirect is never proof of payment.

Future individual tiers and Community Host subscriptions remain separate billing relationships. Enterprise access is manually provisioned and contract-led rather than forced into consumer checkout.

Build noteFeature access is governed by entitlement records and feature flags, not scattered tier checks. Money uses integer minor units plus ISO currency. Provider references and webhook event IDs are unique. Manual processing remains available through an administrator-controlled platform setting if Paystack approval is delayed.

04Data model overview

This is not a full schema — it's the entity map a developer needs to start designing tables/collections without guessing at relationships.

users
Auth record (1:1 with profiles)
profiles
Bio data, links to user_goals (many), interests (many)
communities
Has many community_memberships, posts, events, community_membership_tiers
community_memberships
Join table: user ↔ community, with role (member/moderator/host)
businesses
1 per member who sets one up; has many products_services, reviews
events
Belongs to a community (nullable — platform-level too); has many event_tickets, event_attendees, sponsors
event_tickets / event_attendees
Ticket = purchase record; attendee = attendance/check-in state (kept separate)
orders
Universal transaction record — marketplace items, tickets, or courses via polymorphic item_type/item_id
subscriptions
Recurring billing record; polymorphic across membership tiers and Host billing
courses
Has many enrollments
conversations / messages
Supports DM, group, and community-scoped chat via conversation_type field
travel_listings, opportunity_listings
Content types similar in shape to marketplace listings but domain-specific
notifications
Triggered by nearly every module; own delivery/read-state tracking independent of source events
Relationship pattern to note: Communities, Events, and (eventually) Marketplace listings all share a similar shape — ownership, membership/attendance, monetization, and a feed. Consider whether a shared "hostable entity" abstraction saves duplicated work, versus building each as a fully separate module. This is a genuine architectural trade-off worth discussing with your developer directly rather than deciding in advance here.

07Build phasing

The first release is a 30-day controlled production launch. Daily delivery does not reduce the quality gate: a feature that lacks RLS, failure states, testing or operational ownership stays disabled.

Days 1–3
Foundation & access
Vercel/Supabase environments, migrations, CI, email OTP, temporary beta password, profiles, access states, roles, audit foundations and protected routes. Gate: identity never grants membership or admin authority by itself.
Days 4–7
Onboarding, profiles & member operations
Progressive onboarding, photo, city, languages, professional identity, private contacts, structured interests/goals, consent versions, completeness, visibility pause, admin review and suspension. Gate: pending users cannot enter member data; approved incomplete users reach onboarding.
Days 8–11
Events & content operations
Events, venues, programmes, speakers, announcements, menu, gallery, sponsors, publishing states and event-staff scopes. Gate: staff manage only assigned events and members never see drafts.
Days 12–15
Registration, Paystack & manual mode
Orders, inventory, currencies, automatic/manual/closed modes, signed webhooks, idempotency, references, admin review, receipts, entitlements and reconciliation. Gate: only verified webhook or audited approval grants access once.
Days 16–19
Directory, QR & connections
Event/network directories, search and filters, visibility rules, QR/manual connection code, request/accept/ignore, private-field gates, notes and blocking. Gate: two real accounts connect without exposing private contacts prematurely.
Days 20–22
Messaging, safety & support
Accepted-connection conversations, realtime messages, unread state, report/block flows, report-scoped moderation access, audit logs and support inbox. Gate: admins cannot browse private conversations; escalated access is specific and recorded.
Days 23–27
Ongoing value & controlled extensions
Asks & Offers, post-event feedback, Past Events and notification preferences first. Basic communities or learning may ship only if their security and release gates pass; otherwise remain feature-flagged.
Days 28–30
Hardening & controlled launch
RLS adversarial tests, payment replay/signature checks, mobile/accessibility passes, backups, export/deletion workflow, monitoring, support and incident runbooks, real-account smoke tests and launch sign-off.
Post-launch
Expansion without rework
Full communities and Host toolkit, learning, commercial tiers, marketplace, travel, opportunities/trade, Enterprise and AI. These remain schema-compatible but cannot delay or destabilize the first production loop.

08Confirmed decisions & remaining questions

The following implementation decisions are now fixed for the 30-day release:

  • Authentication: Supabase email OTP is the production method. The beta password is temporary and must be rotated or removed once SMTP passes.
  • Payments: provider-neutral order and entitlement layer; Paystack first; automatic, manual-review and closed/waitlist modes controlled by admin.
  • Client: responsive web/PWA on Vercel first. Native app and app-store distribution require measured post-launch demand.
  • Recommendations: deterministic goal, interest, location and event rules first. AI consumes governed usage data later.
  • Moderation: platform roles plus community-scoped moderators later; private content access is report-triggered and audited.
  • Money: integer minor units and ISO currency codes. No floating-point monetary values and no access granted by browser callbacks.

These questions remain, but none blocks the current foundation:

  • Production domain: confirm the final custom domain and sender subdomain before SMTP and public launch.
  • Legal versions: legal review must approve the Terms, Privacy Notice, Community Guidelines, retention schedule and Kenya data-protection position before public registration.
  • Pricing: confirm Nairobi ticket types, KES prices, standalone membership price and refund/cancellation policy before Day 12.
  • Launch capacity: confirm expected registrations, staff roles, support coverage and check-in device count for realistic load and operational testing.
  • Community portability: define host export and offboarding rules before the post-launch Host toolkit stores community-owned content.
  • Video: select a third-party livestream/replay provider only when an event requires it; custom video infrastructure is excluded.
